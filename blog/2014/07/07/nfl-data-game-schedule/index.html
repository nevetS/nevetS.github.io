NFL Data - Game Schedule Code Snippets

Jul 7th, 2014

The first category of pages to obtain are ones that contain a list of games played for any given week, along with links to the detail pages of those games.

We are downloading each of those pages and storing the HTML source for later analysis.

Here’s how.

Generating a queue

We know the addresses of the pages follow a pattern. The first step is extracting that pattern.

That’s simple enough. We just need to replace {YEAR} with a given year, and {WEEK} with a given week of the regular season.

For this, we can use MessageFormat. I want to be able to configure the pattern just in case I want to re-use part this code later on, so the string itself will go into a properties file.

When using numbers with MessageFormat it will use regional settings to determine whether or not to insert commas or other punctuation into numbers. To avoid that, we can specify a number format within the string.

You will also notice that the commas are escaped. Using PropertiesConfiguration from apache commons, commas are used to delineate lists. This isn’t a list, so I’m escaping the string.

Grabbing the pages

Now that we have the queue put together, how do we grab the pages?

Selenium is pretty easy to work with for simple tasks. In this instance all we really want to do is navigate to a given URL, make sure that the information we want is there, and save the source code.

Selenium uses “WebDrivers” to get the job done. A WebDriver (or driver) is essentially a library that will “drive” a given web browser. There’s a Firefox driver, a Chrome driver, etc. There are even a few headless drivers available.

We’ll use the driver to go to a page, and then tell it to wait until the schedule is available. We do that with a little bit of XPath and WebDriverWait.

The xPath that I configured to look for is:

//div/div/div/div/div/div/div[contains(concat(' '\, normalize-space(@class)\, ' ')\, ' schedules-list ')]

The xPath can be simplified, but the nesting levels increase the speed with which the element I’m looking for is discovered. Additionally, the concat function along with normalize-space takes away any problems associated with similarly defined DOM objects and whitespace definition.

Storing the results

It’s entirely possible to simply follow the links on these pages and store nothing. It makes things a little more complicated to store and analyze later, but I always keep a few things in mind:

  1. If things break partway through, I want as much data as possible to troubleshoot.
  2. Segmenting a project into smaller pieces is good.
  3. There may be additional information on these pages that I want to look at later.

I’m storing the data in mongoDB right out of the gate. Mongo is different than most databases, in fact it doesn’t support SQL at all. It does allow for on the fly generation of what we normally think of as tables, and you don’t have to define a structure. You insert documents and the structure is based on what you insert. That makes things very easy when you want to move quickly. You can always build a strict SQL model later on.
